## Getting Started

1. Read the [`MANIFEST.md`](MANIFEST.md) to understand the philosophy.
2. Read the [`PROTOCOL.md`](PROTOCOL.md) to understand the core protocol.
3. Read the [`FIELD_GUIDE.md`](FIELD_GUIDE.md) to understand how the protocol surfaces.
4. Browse the [open issues](https://github.com/zbbsdsb/Decomposer-Skill/issues) for something to work on.

## What We Accept

- **Protocol improvements**: Corrections, clarifications, or extensions to the five-step protocol.
- **Adapter implementations**: New platform adapters that follow the established pattern.
- **Documentation improvements**: Better examples, clearer explanations, translations.
- **Bug fixes**: Corrections to broken links, rendering issues, or typographical errors.
- **Site improvements**: Enhancements to the GitHub Pages site (`docs/`) that maintain the Swiss Style design language.

## What We Don't Accept

- **Feature requests that add complexity without a clear survival condition**: The protocol is intentionally minimal. Proposals must demonstrate that the addition solves a real failure mode, not just a hypothetical one.
- **Changes that dilute the honesty constraint**: No additions that make the protocol easier to use at the expense of its core honesty mechanism.
- **Rebranding or style changes that deviate from Swiss Style**: The visual identity is intentional. Proposals that add color, gradients, emoji, or decorative elements will be rejected.

## How to Contribute

### 1. Find or Create an Issue

Check existing issues first. If your contribution addresses a problem not yet tracked, create one.

### 2. Discuss the Approach

For significant changes (protocol modifications, new adapters, restructuring), open a discussion in the issue before writing code. This saves time and ensures alignment.

### 3. Fork and Branch

- Fork the repository.
- Create a feature branch: `git checkout -b feat/your-feature-name`.
- Use the prefix `feat/` for features, `fix/` for bug fixes, `docs/` for documentation.

### 4. Make Changes

Follow the [Style Guide](#style-guide).

### 5. Run the Self-Check

Before submitting, run the Anti-Shell Self-Check (from Step 5 of the protocol):

- [ ] Does every node have a survival condition (Alive if / Dead if)?
- [ ] Is the output free of jargon that consumes space without creating knowledge?
- [ ] Is the uncertainty ratio honest (Yellow + Red < 30% is a red flag)?
- [ ] Is there an explicit "I don't know" or "this is outside my scope"?
- [ ] Can the very first concrete step be named in one sentence?

### 6. Submit a Pull Request

See [Pull Request Process](#pull-request-process).

## Development Setup

### Site Development

The GitHub Pages site is a single-page HTML file at `docs/index.html`. No build tools required.

```bash
# Open the site locally
open docs/index.html
```

To preview changes:

1. Edit `docs/index.html` or `docs/demo.html`.
2. Open the file in a browser.
3. Refresh to see changes.

### SVG Assets

The logo and diagram SVGs are at `assets/images/` and `assets/diagrams/`. Edit them directly — they are hand-crafted, not generated by any tool.

## Style Guide

### Protocol Writing

- Use plain English. Avoid jargon, buzzwords, and consultant-speak.
- Every claim must be testable. If it cannot be falsified, it does not belong in the protocol.
- Use the active voice. "The protocol requires" not "It is required by the protocol."
- Keep sentences short. One idea per sentence.

### Code

- **SVG**: Use strict black-and-white. No gradients, no shadows, no fills except where explicitly intended (the center dot of the logo). Use 1px strokes.
- **HTML/CSS**: Follow the Swiss Style design system defined in `docs/index.html`. Use CSS custom properties. No external dependencies. No JavaScript.
- **Markdown**: No emoji. No ASCII art. Use tables and lists for structure.

### Visual Design

The site uses the Swiss Style (International Typographic Style) design system:

- **Color**: Black (#000000), White (#FFFFFF), Gray 50% (#666666), Gray 20% (#CCCCCC).
- **Typography**: Helvetica Neue, Helvetica, Arial, sans-serif.
- **Grid**: 12-column CSS Grid, 8px atomic unit, max-width 1200px.
- **No decoration**: No cards, rounded corners, shadows, gradients, or emoji.

## Pull Request Process

1. Ensure the PR description includes:
   - What problem this solves (and what survival condition it meets).
   - What the change is (one sentence max).
   - What was considered and rejected (if applicable).
2. Ensure the Anti-Shell Self-Check has been run (see step 5 above).
3. Ensure the site renders correctly at 320px, 768px, and 1200px widths.
4. Ensure no emoji or decorative elements were introduced.
5. A maintainer will review within 7 days. If no response, ping the issue.

## Code of Conduct

Be honest. Be direct. Assume good faith. If you disagree with a decision, state your case with evidence, not rhetoric. The protocol is about honest uncertainty, not about being right.
