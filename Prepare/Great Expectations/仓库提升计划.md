# 仓库提升计划（世界级标准 · Swiss Style）

> 目标：把 Great Expectations 从一个「只有 SKILL.md 的仓库」升级为**世界级开源项目门面**——
> 黑白极简的 Swiss / 国际主义排版风格 GitHub Pages、可交互的演示示例、严谨的贡献指南，
> 以及配套治理文件与 CI。标准：以冲击 Lighthouse 100 / WCAG 2.1 AA 为底线，而非目标。

---

## 0. 设计哲学与不可妥协原则

**Swiss / International Typographic Style（国际主义排版）核心法则**
- 数学网格优先：一切元素落在 12 列网格上，不允许"凭感觉"定位。
- 字体即设计：无衬线（Inter / Helvetica Neue），左对齐、右侧参差（ragged-right），不居中堆砌。
- 极致负空间：留白是内容，不是装饰。
- 色彩克制：**黑 (#0A0A0A) / 纸白 (#FAFAFA) 为绝对主色**；瑞士红 (#E30613) 仅作 <5% 强调（链接、激活态、关键标记）。
- 层级靠**字号与字重**，不靠颜色、阴影、圆角。禁用玻璃拟态 / 渐变 / 大圆角——那不是 Swiss。
- 客观性：不堆图表装饰，用排版与规则线（hairline rules）建立节奏。

**正负片切换（Positive / Negative）**：作为对"强制主题切换"规则的瑞士式回应，提供
「正片（黑字白底）/ 负片（白字黑底）」切换——这是 Swiss 的原生二元美学，比浅/深灰主题更高级。

## 2. 设计系统（Design Tokens，单文件 `style.css`）

```
色彩   --ink:#0A0A0A  --paper:#FAFAFA  --line:#E2E2E2  --accent:#E30613
字体   --sans: "Inter","Helvetica Neue",Arial,sans-serif
       --mono: "IBM Plex Mono",ui-monospace,SFMono-Regular,monospace
网格   12 列 / 最大宽 1200px / 槽 24px / 基线 8px
字号（Perfect Fourth 1.333）
       12 / 16 / 21 / 28 / 37 / 50 / 67 / 89 px
间距   4 / 8 / 16 / 24 / 32 / 48 / 64 / 96 / 128 px
规则线 1px hairline (--line)，分隔章节，不靠 box-shadow
动效   仅 IntersectionObserver 淡入/微位移；尊重 prefers-reduced-motion；60fps
```

字体策略：自托管 **Inter 子集 woff2**（仅用到的字形），保证跨平台一致 + 零外部请求。
若求极简，可退回 `system-ui`，但会牺牲一致性——**默认自托管子集**。

## 4. 技术栈与部署（两种方案，默认 A）

**方案 A — 零依赖静态站（推荐，最稳）**
纯 HTML + 现代 CSS + 极小 vanilla JS。无构建、无 npm、无框架依赖。
- 优点：Lighthouse 必满、无构建破损风险、可被任意静态托管、审计友好。
- 部署：仓库 Settings → Pages → 源 = `main` 分支 `/site` 目录；或 GitHub Actions 推 `gh-pages`。
- 维护：多页靠"复制 header/footer"或构建期 include；示例有限，可接受。

**方案 B — Astro（组件化，若页面将膨胀）**
- 零 JS 默认输出（islands），内容用 MDX，部署 GH Pages action。
- 优点：组件复用、示例中文化好；缺点：引入 Node 构建链。

> 决策：以**方案 A 为基线**（符合"最严苛稳健标准"）。若后续页面 >8 或示例 >20，再迁移 Astro。

## 5. 演示示例（Demo）设计

**(a) 精选示例输出（证明 Skill 有效）**
把真实跑通的 Skill 输出（如 Google Glass 回溯测试、西南县 Web3 中心）渲染为
Swiss 风格排版页：左侧"输入"（目标/共识/稀缺资源），右侧"八段输出"。这是 repo 的"王牌证据"。

**(b) 交互式 Prompt 组装器（demo.html）**
- 表单三字段：Goal / 最大共识 / 最稀缺资源。
- 点击"Compose" → 前端把 `SKILL.md` 系统指令 + 用户输入拼成**可直接复制的完整 Prompt**，
  并附"打开 ChatGPT/Claude"按钮（链接带 `?` 或复制）。
- 诚实边界：静态站无 LLM key，**不伪造生成**；组装器只产出"喂给任意 LLM 的提示词"。
- 展示区：内嵌 (a) 的示例作为"看它长什么样"。

## 6. 贡献指南（CONTRIBUTING）设计

- **哲学对齐**：先讲清"反共识 + 完备性"双核，避免贡献者把 Skill 改回平庸。
- **如何改提示词**：fork → 编辑 `SKILL.md` → 必须跑一次回溯测试（提供测试用例）→ 在 PR 贴出前后对比。
- **示例/测试用例规范**：`examples/` 命名 `topic.md`，含标准三段输入 + 八段输出。
- **SKILL.md 风格红线**：frontmatter 合法、禁用词库不可删、强制八段结构不可省、量化指标不可降。
- **Definition of Done**：假设颠覆率 ≥50%、完备性通过率 >80%、可证伪断言 ≥3、PESTEL 零缺项。
- **PR / Issue 模板**：变更类型、影响路径、测试用例、验收截图。

## 10. 分阶段路线图

- **P0 设计系统**：token、grid、type scale、正负片切换、favicon/og → 验收：风格指南页可通过上述视觉门禁。
- **P1 核心页**：index（Manifesto）+ skill.html（四支柱/工作流） → 验收：Lighthouse 四项 100。
- **P2 演示**：examples/ 真实输出 + demo.html 组装器 → 验收：示例可渲染、组装器复制可用、无 JS 报错。
- **P3 治理**：CONTRIBUTING / LICENSE / CoC / 模板 / README 重写 → 验收：外部贡献者可按文档提 PR。
- **P4 CI/CD**：deploy + docs(链接/HTML 校验) → 验收：死链/校验失败自动阻断。
- **P5 打磨与发布**：a11y/perf 终检、og/favicon 上线、站点公告 → 验收：全部门禁通过。

**合计估算：约 6–8 个工作日**（设计系统占大头）。

## 11. 风险与对策

| 风险 | 对策 |
|------|------|
| 极简导致信息密度低、看不出"世界级" | 用强排版节奏 + 规则线 + 大字号对比弥补，而非加装饰 |
| 纯静态难以维护多页 | 示例有限；若膨胀再迁 Astro（方案 B） |
| 演示"伪造生成"违背诚信 | 组装器只产出可复制 Prompt，明确标注"需自备 LLM" |
| a11y 被极简美学牺牲（对比度/焦点） | 红仅作强调，正文严格 ≥4.5:1；焦点环强制可见 |
| 负片模式对比度翻车 | 切换时重算 token，CI 加对比度检查 |
